## Related Panel Bug Fixes — CloudTrail Events, IAM Policies, Role Navigation

### CloudTrail Events Counter Mismatch (Bug 1)

The related panel for IAM Users and EC2 instances was showing a misleading count for **CloudTrail Events** (e.g., `(4)`) when the event cache only covered the first page of results. Navigating would then reveal 50+ events — a clear mismatch.

**Fix:** When the event cache is truncated, the counter is omitted and navigation triggers a server-side filtered fetch (`LookupAttributes` with `Username` or `ResourceName`), which returns only matching events across all pages. Load-more (`m`) continues to page through filtered results correctly.

### IAM Policies Empty List (Bug 2)

Navigating from an IAM User, IAM Role, or IAM Group to **IAM Policies** was opening an empty list even when the related panel showed `(1)` or more.

**Fix:** The policy ID comparison was using the ARN (`arn:aws:iam::...`) while the policy fetcher stores the policy name as the resource ID. All three checkers (user, role, group) now use `PolicyName` for matching.

### CloudTrail AssumedRole — IAM Roles Show (0) (Bug 3)

For CloudTrail events generated by assumed-role sessions, the **IAM Roles** related panel always showed `(0)`. The role name is encoded in the raw event JSON (`userIdentity.sessionContext.sessionIssuer.userName`) rather than in the `Resources` list.

**Fix:**
- The event fetcher now parses the `CloudTrailEvent` JSON string and stores the role name in `Fields["role_name"]`
- `ctEventsExtractRoleName` gains a third extraction path for AssumedRole events
- `ct-events` registers `user → iam-user` and `role_name → role` as navigable fields — usernames and role names are now clickable in the detail view
