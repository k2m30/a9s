## CloudTrail Events list redesign + codebase concerns audit

### CloudTrail events v2

The list view is rebuilt around three colors. Every row gets exactly one tint based on a severity ladder, and every cell — verb glyph, actor, target, time — inherits it. No more per-cell ANSI composition.

| Tier | Color | Triggers |
|---|---|---|
| `ct-info` | dim grey | routine reads — normal-volume noise |
| `ct-attention` | yellow | writes, ROOT, sensitive reads, cross-account |
| `ct-danger` | red | destructive ops or failures |

Other ct-events changes:
- **AssumeRole / AssumeRoleWithSAML reclassified `W` → `R`** — STS session vending is identity exchange, not mutation. Ordinary AssumeRole events drop from `ct-attention` to `ct-info`; cross-account, Root, and errors still escalate. (`AssumeRoleWithWebIdentity` was already `R`.)
- **TIME column** now `Apr 07 17:00:59` (15 chars) instead of ISO timestamp.
- **TARGET column** strips ARN prefix to show just the resource portion.
- **ACTOR column** shows counterparty account inline for cross-account events: `999988887777/alice`.
- **Cross-account TARGET ARNs** annotated the same way.
- **Sort indicator glyph** bound to exactly one column per sort mode (fixes double-glyph on TIME/EVENT).
- **Sensitive-reads allowlist** escalates Secrets Manager, SSM Parameters, STS session-vending, IAM credential reports, and ACM exports to `ct-attention`.
- **Target fallback table** for management events with empty `resources[]`.
- **Help legend** rewritten — shows the three actual row-tint colors instead of a seven-color decorative palette that contradicted the design spec.

### `ctrl+z` — attention-only filter

A global "show only attention-worthy rows" toggle on every resource list view. Hides `ct-info` events, terminated EC2 instances, and other dim rows so you only see what's worth a glance.

### App-wide cancellation context

`tui.Model` now owns an `appCtx` created in `New()` and cancelled on quit. Fetchers and IAM related-checkers use it instead of synthesizing fresh `context.Background()` per call. Quitting a9s — or, in a follow-up release, navigating away from a slow IAM detail — actually cancels the in-flight AWS calls instead of letting them complete and deliver to a dead view.

### IAM `ListEntitiesForPolicy` consolidation

Policy detail views previously made three filtered `ListEntitiesForPolicy` calls (users, groups, roles) per related-fan-out cycle. Now: one unfiltered call partitioned client-side, with a 5-second per-policy cache. **3× fewer IAM API calls** on policy detail.

### Demo-mode bug fixes

- **Related counter mismatch fixed.** Opening a `policy` detail in demo mode used to show "5 Roles" while pressing Enter opened an empty list. Affected `policy → role/iam-user/iam-group` and `role → lambda/glue` — counters now match the actual fixture IDs.
- Demo HTTP handlers no longer swallow `Sscanf`/`Unmarshal` errors — malformed pagination tokens produce HTTP 400 instead of zero-offset silent success.
- Demo fixture `mustParseTime` panic replaced with `ParseTime` returning an error; new smoke test exercises every fixture init().

### Codebase concerns audit (waves A+B+C)

Triaged `.planning/codebase/CONCERNS.md` and landed the high-value fixes:

- `cache.Load` validates resource keys against the registry — unknown keys are skipped, not silently retained.
- Removed deprecated `SecretName` field from `ValueRevealedMsg` (internal cleanup; all callers now use `ResourceID`).
- New regression-guard tests (+200 tests, total 11,893):
  - registry init-order validation (every child type has a parent, every type has a fetcher)
  - cache key validation
  - demo fixture load smoke test
  - demo webhook URL scanner (catches accidental real Slack/Discord/PagerDuty endpoints)
  - pagination `loadingMore` error/double-load/pop guards
  - app cancellation propagation
  - cross-account actor format
