## a9s v3.3.0

### New Child Views: ECS Services

Three new child views for ECS Services -- the multi-child pattern gives you three different lenses on every service:

- **ECS Services → Tasks** (`Enter`) -- Running and recently stopped tasks with status, health check result, task definition revision, and stopped reason. The first thing to check during a deployment or crash loop. Fetches both RUNNING and STOPPED tasks, batched in chunks of 100 per AWS API limits.
- **ECS Services → Service Events** (`e`) -- Event timeline showing steady state confirmations, placement failures, target group registrations, and deployment progress. Data comes from DescribeServices (no extra API call).
- **ECS Services → Container Logs** (`L`) -- Application logs from CloudWatch, resolved automatically from the task definition's awslogs configuration. Cross-service view: reads the task definition to find the log group, then fetches recent log events. Shows container name and short task ID in the Stream column.

### How it works

ECS is the first **multi-child parent** in a9s. Each key opens a different child view from the same parent list:
- `Enter` = Tasks (what's running?)
- `e` = Events (what happened?)
- `L` = Logs (what's the app outputting?)

Container Logs is a **cross-service** view -- it calls ECS DescribeTaskDefinition to resolve the CloudWatch log group, then calls CloudWatch Logs FilterLogEvents. If the task definition uses a non-awslogs driver (e.g., firelens), a clear error message is shown.

### Tooling

- New `cmd/viewsgen/` tool auto-generates `.a9s/views.yaml` from built-in defaults. Run `go run ./cmd/viewsgen/ > .a9s/views.yaml` to regenerate.
- `.a9s/views.yaml` now includes all 71 resource type definitions (was missing 30 previously).
- Round-trip test ensures views.yaml stays in sync with defaults.go.
