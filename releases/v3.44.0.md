## Partial-Success Contract End-to-End + tui Package Split

### Partial results survive per-item failures everywhere

Every layer that calls AWS — paginated list fetchers, the synchronous demo / no-cache prefetch, the per-type availability probe, and Wave-2 enrichers — now preserves the partial slice when an error accompanies it. Previously a single per-item failure (one throttled `DescribeTable`, one failed `ListGroupPolicies`) blanked the entire menu badge or list view for that resource type.

The fix:

- `ResourcesLoadedMsg`, `AvailabilityCheckedMsg`, and `EnrichmentCheckedMsg` carry the partial slice alongside `Err` (composite error from `AggregateFailures`).
- `fetchResources` / `fetchMoreResources` / `probeResourceAvailability` / `probeEnrichment` only emit a hard `APIErrorMsg` when there are zero resources to keep; otherwise they keep the resources and propagate the error.
- `handleAvailabilityChecked` / `handleEnrichmentChecked` apply state from the partial slice AND emit a `FlashMsg{IsError:true}` so the `!` log shows the operator what failed.

### Lazy-add cache no longer suppresses real fetches

A lazy-only cache entry (sparse rows pulled by `FetchByIDs` for AWS-managed KMS keys, public AMIs, shared snapshots, AWS-managed IAM policies) is now marked `IsTruncated:true` in the snapshot. The related-panel prefetch decision uses a `mainCacheKeys` set built from `resourceCache` only — so a lazy-only KMS entry no longer suppresses the real first-page KMS fetch when an EC2 → KMS reverse-scan checker runs.

The lazy-only fast-path during related navigation now requires ALL requested IDs to short-circuit: `len(filtered) == len(result.RelatedIDs)`. Partial coverage (some IDs missing) falls through to a full fetch instead of rendering only the lazily-cached subset.

### IAM policy lookup also indexes by ARN

`FetchIAMPoliciesByIDsFull` now keys cached entries by both `PolicyName` and `Arn`, so a checker that emits the role → managed-policy ARN form drills into a real entry instead of landing on an empty list.

### tui package split for the 500-line budget

The `internal/tui` package was reorganized so every file fits under the file-size budget defined in `docs/go-codebase-checklist.md`. New files:

- `app_probes.go` — `probeResourceAvailability`, `probeEnrichment`, `loadAvailabilityCache`, `saveAvailabilityCache`, `demoPrefetchCounts`, `buildEnrichQueue`, `refreshResourceListWithEnrichmentRerun`.
- `app_handlers_availability.go` — `handleAvailabilityCacheLoaded`, `handleAvailabilityPrefetched`, `handleAvailabilityChecked`, `handleEnrichmentChecked`, `startEnrichment`, `unifiedIssueCount`.
- `app_handlers_related_navigate.go` — `handleRelatedNavigate` and `handleRelatedNavigateChild`.
- `related_cache.go` — `relatedCacheLRU`, `relatedCacheKey`, `relatedCacheReplay`.

`app_fetchers.go`, `app_handlers_navigate.go`, and `app_related.go` shrunk by 300+ lines each. Behavior unchanged.

### Defensive nil-clients guard restored

`probeEnrichment` had its `clients == nil` guard removed during a partial-success refactor, leaking a nil pointer into Wave-2 enrichers like `EnrichEC2InstanceStatus`. Restored — Wave 2 now returns a clear "AWS clients not initialized" error instead of crashing.

### Smaller fixes

- `FetchAMIsByIDs` now sets `IncludeDeprecated: true` (matches `FetchAMIByID`) so deprecated AMIs referenced from EC2 / ASG / EKS pivots survive batch drills.
- `efs_issue_enrichment` uses the typed `efstypes.LifeCycleStateAvailable` constant.
