### `athena`

- **Why related**: Athena queries read/write against S3 ‚Äî this bucket may be the data source for scans or the query-result output location.
- **How discovered**: Reverse-scan the already-loaded `athena` list ‚Äî inspect each workgroup's `ResultConfiguration.OutputLocation` (an `s3://bucket/prefix` URI) and match the bucket segment against this bucket's `Name`. Data-source buckets referenced by user queries are not on the workgroup object and are out of scope. ‚Äî a9s-devops: workgroup output-location is the only AWS-surface link; table-level data-source mapping requires Glue Data Catalog walking which is Wave 3.
- **Count shown**: yes.

### `backup`

- **Why related**: This bucket may be protected by an AWS Backup plan ‚Äî operator wants to confirm the bucket is covered before declaring a recovery posture.
- **How discovered**: Call `backup:ListProtectedResourcesByBackupVault` or scan the already-loaded `backup` list's selections (`ListBackupSelections` ‚Üí `GetBackupSelection.Resources[]`) for `arn:aws:s3:::<bucket-name>`. ‚Äî a9s-devops: Backup coverage lives on the plan's selection, not on the bucket. Reverse-scan against the already-loaded `backup` list is the cheapest approach for a9s.
- **Count shown**: yes.

### `cf`

- **Why related**: CloudFront distributions may serve this bucket as an origin ‚Äî investigating viewer-facing latency or 403s starts from the bucket and jumps to the distribution.
- **How discovered**: Reverse-scan the already-loaded `cf` list ‚Äî for each distribution, walk `DistributionConfig.Origins.Items[].DomainName` and match `<bucket>.s3.<region>.amazonaws.com` or `<bucket>.s3-website-<region>.amazonaws.com` against this bucket's `Name`. ‚Äî a9s-devops: origin list is the definitive field; cite AWS API Reference API_Origin.DomainName.
- **Count shown**: yes.

### `cfn`

- **Why related**: The CloudFormation stack that provisioned this bucket is the fastest route to the template, parameters, and stack events ‚Äî critical for drift and change history.
- **How discovered**: Read the bucket's `aws:cloudformation:stack-name` tag (set automatically on CFN-managed resources) via `s3:GetBucketTagging` and look up the stack in the already-loaded `cfn` list. ‚Äî a9s-devops: the tag is the canonical CFN-managed marker; no bucket tag ‚Üí not CFN-managed (skip).
- **Count shown**: yes.

### `eb-rule`

- **Why related**: EventBridge rules may fire on S3 object events (created/deleted) ‚Äî this bucket may be the source triggering downstream automation.
- **How discovered**: Reverse-scan the already-loaded `eb-rule` list ‚Äî rules with `EventPattern.source=["aws.s3"]` AND `EventPattern.detail.bucket.name` matching this bucket. ‚Äî a9s-devops: event-pattern field is standard for S3-originated EventBridge rules; cite AWS Events User Guide ¬ßS3 event notifications.
- **Count shown**: yes.

### `glue`

- **Why related**: Glue crawlers populate the Data Catalog from S3 ‚Äî operator debugging a stale table wants to hop to the crawler.
- **How discovered**: Call `glue:GetCrawlers` and inspect each crawler's `Targets.S3Targets[].Path` for a prefix matching `s3://<bucket>`. This is a separate API (not on the bucket). ‚Äî a9s-devops: S3 targets live on the crawler, not on the bucket; `GetCrawlers` is account-wide and cheap.
- **Count shown**: yes.

### `iam-user`

- **Why related**: Audit pivot ‚Äî when triaging a bucket-access incident operator wants to see the IAM users that appear on principal-level access logs or ownership (e.g. `Owner.DisplayName`). ‚Äî a9s-devops: not worth it, see ¬ß5 Out of Scope.
- **How discovered**: `TBD ‚Äî a9s-devops: not available in AWS surface for an operator-grade pivot; bucket-level principal attribution requires CloudTrail data-plane parsing, which is Wave 3. Owner is an account ID, not a user.`
- **Count shown**: unknown.

### `kms`

- **Why related**: Bucket is encrypted with a customer-managed KMS key via SSE-KMS ‚Äî operator wants to confirm the key is enabled and not pending deletion before declaring the bucket usable.
- **How discovered**: Call `s3:GetBucketEncryption` and read `ServerSideEncryptionConfiguration.Rules[].ApplyServerSideEncryptionByDefault.KMSMasterKeyID`; look the ARN up in the already-loaded `kms` list. ‚Äî a9s-devops: `GetBucketEncryption` is the canonical source; AWS-managed `aws/s3` keys map to a standard alias ‚Äî treat them as "managed default" in the panel.
- **Count shown**: yes.

### `lambda`

- **Why related**: Lambda functions may be subscribed to S3 object events on this bucket ‚Äî operator chasing an event-driven failure starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `LambdaFunctionConfigurations[].LambdaFunctionArn`; look the ARN up in the already-loaded `lambda` list. ‚Äî a9s-devops: notification configuration is the definitive bucket-side source; reverse-scanning lambda event-source-mappings will miss S3-as-source because ESMs are for Kinesis/SQS/DynamoDB.
- **Count shown**: yes.

### `logs`

- **Why related**: S3 server-access logs are delivered to another S3 bucket, not CloudWatch Logs ‚Äî the `logs` panel entry is for buckets *receiving* this bucket's access logs (reverse pivot) and for buckets that ship *their* logs here. The per-type hint `"Server access-log target bucket"` is a cross-bucket S3‚ÜíS3 link, not an S3‚ÜíCloudWatch link.
- **How discovered**: Call `s3:GetBucketLogging` ‚Üí `LoggingEnabled.TargetBucket`; if set, the target is another S3 bucket. For the reverse direction, scan the already-loaded `s3` list for any bucket whose logging target equals this bucket. ‚Äî a9s-devops: S3 access logs never go to CloudWatch Logs; the `logs` entry in related-resources is misleading shorthand for "log-destination bucket". Worth surfacing ‚Äî operators chasing retention/cost/forensics hit this regularly.
- **Count shown**: yes.

### `r53`

- **Why related**: A Route 53 alias record may point to this bucket's website endpoint ‚Äî chasing a `NoSuchBucket` or 404 for a custom domain lands here.
- **How discovered**: Reverse-scan the already-loaded `r53` list ‚Äî for each hosted zone, list `ResourceRecordSets` where `AliasTarget.DNSName` matches `s3-website-<region>.amazonaws.com.` AND the record name matches this bucket's `Name` (website-endpoint convention requires bucket name == FQDN). ‚Äî a9s-devops: alias to S3 requires bucket-name==FQDN; that's the join key. Cite AWS Route 53 Developer Guide ¬ßRouting-traffic-to-website.
- **Count shown**: yes.

### `role`

- **Why related**: Audit pivot ‚Äî when the bucket policy references an `AWS` principal that is an IAM role ARN, operator wants to hop to that role. ‚Äî a9s-devops: possible but low value for a daily-driver ‚Äî bucket policies are typically maintained in IaC and the role link is better discovered from the role side.
- **How discovered**: Call `s3:GetBucketPolicy`, parse the JSON policy document for `Statement[].Principal.AWS` entries matching IAM role ARNs, look each up in the already-loaded `role` list. ‚Äî a9s-devops: possible via bucket-policy parsing but fragile (wildcard principals, service principals, cross-account IDs) ‚Äî worth keeping as an advanced pivot, not a first-order one.
- **Count shown**: yes.

### `sns`

- **Why related**: Bucket event notifications may publish to an SNS topic ‚Äî operator chasing a fan-out notification path starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `TopicConfigurations[].TopicArn`; look the ARN up in the already-loaded `sns` list. ‚Äî a9s-devops: same single API as Lambda notifications; one call covers Lambda + SNS + SQS destinations.
- **Count shown**: yes.

### `sqs`

- **Why related**: Bucket event notifications may enqueue to an SQS queue ‚Äî operator chasing a stuck consumer or backlog starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `QueueConfigurations[].QueueArn`; look the ARN up in the already-loaded `sqs` list. ‚Äî a9s-devops: same single-call source as SNS/Lambda notifications.
- **Count shown**: yes.

### `trail`

- **Why related**: This bucket may be a destination for one or more CloudTrail trails ‚Äî operator investigating a trail-delivery failure jumps to the trails targeting this bucket.
- **How discovered**: Reverse-scan the already-loaded `trail` list ‚Äî match `Trail.S3BucketName == this.Name`. ‚Äî a9s-devops: `S3BucketName` is the canonical field on `DescribeTrails`; reverse-scan is one pass over the trail list with zero extra API calls.
- **Count shown**: yes.

### `waf`

- **Why related**: Audit pivot ‚Äî a WAF web ACL does not directly attach to an S3 bucket (ACLs attach to CloudFront, ALB, API Gateway, AppSync, Cognito). The link is indirect, via a CloudFront distribution fronting this bucket. ‚Äî a9s-devops: not worth it as a first-order S3 pivot; better reached through `cf`. See ¬ß5 Out of Scope.
- **How discovered**: `TBD ‚Äî a9s-devops: not available in AWS surface as a direct S3‚ÜíWAF edge; the only path is S3‚ÜíCloudFront‚ÜíWAF which is already reachable via the `cf` panel entry.`
- **Count shown**: unknown.

## 3. Attention / Issues Algorithm

Transcribed from `docs/attention-signals.md`.

### 3.1 Wave 1 ‚Äî zero extra API calls

No Wave 1 signals ‚Äî the list API does not return fields usable for attention. Per the attention-signals.md row, `ListBuckets` returns `Name`, `CreationDate`, `BucketRegion`, `BucketArn` only. There is no lifecycle / state / health field to classify from.

## 4. Issue Visualization

Every signal from ¬ß3.1 and ¬ß3.2 must land on one or more of these five existing surfaces. No other UI is allowed.

| # | Surface | Mechanism |
|---|---|---|
| S1 | Menu `issues:N` count | Aggregated count of `!`-severity findings. `~` findings do not bump. |
| S2 | Row color (list view) | Row colored by state bucket ‚Äî Healthy=green, Warning=yellow, Broken=red, Dim=gray. Yellow/red/dim are themselves the attention signal. |
| S3 | `!` / `~` glyph before the name | Annotates a Healthy (green) row with "no immediate action, but worth knowing" ‚Äî e.g. maintenance scheduled, certificate expiring soon. `!` = important background concern, `~` = informational. **Never appears on yellow/red/dim rows.** |
| S4 | Status / description column text | Short human-readable cause (e.g. `public access block missing`). **Healthy rows render blank** ‚Äî no `OK` / `available` / `ACTIVE` / `running`. Empty means "nothing to see." |

## 4.1 UX review

At 3am, glancing at the list, can the operator tell what's wrong with a problem row without opening detail? Yes for this resource ‚Äî any bucket flagged with `!` carries `public access block incomplete` in the Status column, which names the concern plainly. The detail line adds the single nuance an operator needs (account-level PAB may override) without forcing a navigation. All problem rows are self-explanatory in the list ‚Äî operator can triage without opening detail.
