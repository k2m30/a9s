# s3 — Resource Spec

Golden UX/UI doc for this resource, written from the operator's perspective. Describes what the list row, Status column, glyphs, and detail view should look like — the should-be, not the is. Implementation conforms to this doc; tests assert against it. When code and this doc disagree, the code is wrong.

## 1. Identity

- **shortName**: `s3`
- **Display name**: S3 Buckets
- **AWS API reference**: https://docs.aws.amazon.com/AmazonS3/latest/API/API_ListBuckets.html
- **List API**: `ListBuckets` (returns per-bucket `Name`, `CreationDate`, `BucketRegion`, `BucketArn` only — no state field).
- **Describe API (if any)**: `GetPublicAccessBlock` (per bucket; used in Wave 2).

## 2. Related Resources Panel (detail view, right column)

Expected targets from `docs/related-resources.md` Per-type contract: `athena`, `backup`, `cf`, `cfn`, `eb-rule`, `glue`, `iam-user`, `kms`, `lambda`, `logs`, `r53`, `role`, `sns`, `sqs`, `trail`, `waf`, `ct-events`.

**Interpretation note for `logs`**: the contract lists `logs` but for S3 this does NOT mean CloudWatch Log Groups — S3 server-access logs are delivered to another S3 bucket. a9s therefore registers this pivot as `s3` (DisplayName "Access Log Bucket"), not `logs`. See the `logs` subsection below for full reasoning and the a9s-devops citation.

### `athena`

- **Why related**: Athena queries read/write against S3 — this bucket may be the data source for scans or the query-result output location.
- **How discovered**: Reverse-scan the already-loaded `athena` list — inspect each workgroup's `ResultConfiguration.OutputLocation` (an `s3://bucket/prefix` URI) and match the bucket segment against this bucket's `Name`. Data-source buckets referenced by user queries are not on the workgroup object and are out of scope. — a9s-devops: workgroup output-location is the only AWS-surface link; table-level data-source mapping requires Glue Data Catalog walking which is Wave 3.
- **Count shown**: yes.

### `backup`

- **Why related**: This bucket may be protected by an AWS Backup plan — operator wants to confirm the bucket is covered before declaring a recovery posture.
- **How discovered**: Call `backup:ListProtectedResourcesByBackupVault` or scan the already-loaded `backup` list's selections (`ListBackupSelections` → `GetBackupSelection.Resources[]`) for `arn:aws:s3:::<bucket-name>`. — a9s-devops: Backup coverage lives on the plan's selection, not on the bucket. Reverse-scan against the already-loaded `backup` list is the cheapest approach for a9s.
- **Count shown**: yes.

### `cf`

- **Why related**: CloudFront distributions may serve this bucket as an origin — investigating viewer-facing latency or 403s starts from the bucket and jumps to the distribution.
- **How discovered**: Reverse-scan the already-loaded `cf` list — for each distribution, walk `DistributionConfig.Origins.Items[].DomainName` and match `<bucket>.s3.<region>.amazonaws.com` or `<bucket>.s3-website-<region>.amazonaws.com` against this bucket's `Name`. — a9s-devops: origin list is the definitive field; cite AWS API Reference API_Origin.DomainName.
- **Count shown**: yes.

### `cfn`

- **Why related**: The CloudFormation stack that provisioned this bucket is the fastest route to the template, parameters, and stack events — critical for drift and change history.
- **How discovered**: Read the bucket's `aws:cloudformation:stack-name` tag (set automatically on CFN-managed resources) via `s3:GetBucketTagging` and look up the stack in the already-loaded `cfn` list. — a9s-devops: the tag is the canonical CFN-managed marker; no bucket tag → not CFN-managed (skip).
- **Count shown**: yes.

### `eb-rule`

- **Why related**: EventBridge rules may fire on S3 object events (created/deleted) — this bucket may be the source triggering downstream automation.
- **How discovered**: Reverse-scan the already-loaded `eb-rule` list — rules with `EventPattern.source=["aws.s3"]` AND `EventPattern.detail.bucket.name` matching this bucket. — a9s-devops: event-pattern field is standard for S3-originated EventBridge rules; cite AWS Events User Guide §S3 event notifications.
- **Count shown**: yes.

### `glue`

- **Why related**: Glue crawlers populate the Data Catalog from S3 — operator debugging a stale table wants to hop to the crawler.
- **How discovered**: Call `glue:GetCrawlers` and inspect each crawler's `Targets.S3Targets[].Path` for a prefix matching `s3://<bucket>`. This is a separate API (not on the bucket). — a9s-devops: S3 targets live on the crawler, not on the bucket; `GetCrawlers` is account-wide and cheap.
- **Count shown**: yes.

### `iam-user`

- **Why related**: Audit pivot — when triaging a bucket-access incident operator wants to see the IAM users that appear on principal-level access logs or ownership (e.g. `Owner.DisplayName`). — a9s-devops: not worth it, see §5 Out of Scope.
- **How discovered**: `TBD — a9s-devops: not available in AWS surface for an operator-grade pivot; bucket-level principal attribution requires CloudTrail data-plane parsing, which is Wave 3. Owner is an account ID, not a user.`
- **Count shown**: unknown.

### `kms`

- **Why related**: Bucket is encrypted with a customer-managed KMS key via SSE-KMS — operator wants to confirm the key is enabled and not pending deletion before declaring the bucket usable.
- **How discovered**: Call `s3:GetBucketEncryption` and read `ServerSideEncryptionConfiguration.Rules[].ApplyServerSideEncryptionByDefault.KMSMasterKeyID`; look the ARN up in the already-loaded `kms` list. — a9s-devops: `GetBucketEncryption` is the canonical source; AWS-managed `aws/s3` keys map to a standard alias — treat them as "managed default" in the panel.
- **Count shown**: yes.

### `lambda`

- **Why related**: Lambda functions may be subscribed to S3 object events on this bucket — operator chasing an event-driven failure starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `LambdaFunctionConfigurations[].LambdaFunctionArn`; look the ARN up in the already-loaded `lambda` list. — a9s-devops: notification configuration is the definitive bucket-side source; reverse-scanning lambda event-source-mappings will miss S3-as-source because ESMs are for Kinesis/SQS/DynamoDB.
- **Count shown**: yes.

### `logs` (implemented as `s3` — "Access Log Bucket")

- **Why related**: S3 server-access logs are delivered to another S3 bucket, not CloudWatch Logs. The `logs` entry in the per-type contract is misleading shorthand for "log-destination bucket" — in the running app this pivot is registered with `TargetType="s3"` and `DisplayName="Access Log Bucket"` so navigation lands on the destination bucket's S3 row, not on an empty CloudWatch Log Groups list.
- **How discovered**: Call `s3:GetBucketLogging` → `LoggingEnabled.TargetBucket`; the target IS another S3 bucket. Emitted by `checkS3Logs` as `TargetType="s3"` with the destination bucket name as the navigation ID. Reverse direction (buckets shipping logs HERE) is not currently implemented. — a9s-devops: S3 access logs never go to CloudWatch Logs. Worth surfacing — operators chasing retention/cost/forensics hit this regularly.
- **Count shown**: yes.

### `r53`

- **Why related**: A Route 53 alias record may point to this bucket's website endpoint — chasing a `NoSuchBucket` or 404 for a custom domain lands here.
- **How discovered**: Reverse-scan the already-loaded `r53` list — for each hosted zone, list `ResourceRecordSets` where `AliasTarget.DNSName` matches `s3-website-<region>.amazonaws.com.` AND the record name matches this bucket's `Name` (website-endpoint convention requires bucket name == FQDN). — a9s-devops: alias to S3 requires bucket-name==FQDN; that's the join key. Cite AWS Route 53 Developer Guide §Routing-traffic-to-website.
- **Count shown**: yes.

### `role`

- **Why related**: Audit pivot — when the bucket policy references an `AWS` principal that is an IAM role ARN, operator wants to hop to that role. — a9s-devops: possible but low value for a daily-driver — bucket policies are typically maintained in IaC and the role link is better discovered from the role side.
- **How discovered**: Call `s3:GetBucketPolicy`, parse the JSON policy document for `Statement[].Principal.AWS` entries matching IAM role ARNs, look each up in the already-loaded `role` list. — a9s-devops: possible via bucket-policy parsing but fragile (wildcard principals, service principals, cross-account IDs) — worth keeping as an advanced pivot, not a first-order one.
- **Count shown**: yes.

### `sns`

- **Why related**: Bucket event notifications may publish to an SNS topic — operator chasing a fan-out notification path starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `TopicConfigurations[].TopicArn`; look the ARN up in the already-loaded `sns` list. — a9s-devops: same single API as Lambda notifications; one call covers Lambda + SNS + SQS destinations.
- **Count shown**: yes.

### `sqs`

- **Why related**: Bucket event notifications may enqueue to an SQS queue — operator chasing a stuck consumer or backlog starts here.
- **How discovered**: Call `s3:GetBucketNotificationConfiguration` and read `QueueConfigurations[].QueueArn`; look the ARN up in the already-loaded `sqs` list. — a9s-devops: same single-call source as SNS/Lambda notifications.
- **Count shown**: yes.

### `trail`

- **Why related**: This bucket may be a destination for one or more CloudTrail trails — operator investigating a trail-delivery failure jumps to the trails targeting this bucket.
- **How discovered**: Reverse-scan the already-loaded `trail` list — match `Trail.S3BucketName == this.Name`. — a9s-devops: `S3BucketName` is the canonical field on `DescribeTrails`; reverse-scan is one pass over the trail list with zero extra API calls.
- **Count shown**: yes.

### `waf`

- **Why related**: Audit pivot — a WAF web ACL does not directly attach to an S3 bucket (ACLs attach to CloudFront, ALB, API Gateway, AppSync, Cognito). The link is indirect, via a CloudFront distribution fronting this bucket. — a9s-devops: not worth it as a first-order S3 pivot; better reached through `cf`. See §5 Out of Scope.
- **How discovered**: `TBD — a9s-devops: not available in AWS surface as a direct S3→WAF edge; the only path is S3→CloudFront→WAF which is already reachable via the `cf` panel entry.`
- **Count shown**: unknown.

### `ct-events`

- **Why related**: Universal pivot — applies to every registered type; see `related-resources.md` §Policy.
- **How discovered**: CloudTrail `LookupEvents` with `LookupAttribute=ResourceName` = bucket name (and/or `ResourceARN` = `arn:aws:s3:::<bucket>`).
- **Count shown**: yes.

## 3. Attention / Issues Algorithm

Transcribed from `docs/attention-signals.md`.

### 3.1 Wave 1 — zero extra API calls

No Wave 1 signals — the list API does not return fields usable for attention. Per the attention-signals.md row, `ListBuckets` returns `Name`, `CreationDate`, `BucketRegion`, `BucketArn` only. There is no lifecycle / state / health field to classify from.

### 3.2 Wave 2 — bounded extra API calls

- **Signal**: `GetPublicAccessBlock` per bucket returns `NoSuchPublicAccessBlockConfiguration` error OR any of `BlockPublicAcls` / `IgnorePublicAcls` / `BlockPublicPolicy` / `RestrictPublicBuckets` is `false`.
  - **State bucket**: Healthy (the row stays green — S3 has no Wave 1 color signal; the finding is a background-check annotation, not a primary row state).
  - **API call**: `GetPublicAccessBlock` — one per bucket.
  - **Cost shape**: per-resource.

### 3.3 Wave 3 — OUT OF SCOPE

- OUT OF SCOPE: `GetBucketEncryption`
- OUT OF SCOPE: `GetBucketVersioning`
- OUT OF SCOPE: `GetBucketLogging`
- OUT OF SCOPE: `GetBucketLifecycleConfiguration` (each is per-bucket)

## 4. Issue Visualization

Every signal from §3.1 and §3.2 must land on one or more of these five existing surfaces. No other UI is allowed.

| # | Surface | Mechanism |
|---|---|---|
| S1 | Menu `issues:N` count | Aggregated count of `!`-severity findings. `~` findings do not bump. |
| S2 | Row color (list view) | Row colored by state bucket — Healthy=green, Warning=yellow, Broken=red, Dim=gray. Yellow/red/dim are themselves the attention signal. |
| S3 | `!` / `~` glyph before the name | Annotates a Healthy (green) row with "no immediate action, but worth knowing" — e.g. maintenance scheduled, certificate expiring soon. `!` = important background concern, `~` = informational. **Never appears on yellow/red/dim rows.** |
| S4 | Status / description column text | Short human-readable cause (e.g. `public access block missing`). **Healthy rows render blank** — no `OK` / `available` / `ACTIVE` / `running`. Empty means "nothing to see." |
| S5 | Detail view enrichment line | Short operator-readable sentence rendered inline in the detail view. No ceremonial header. |

Wave → surface mapping applied here:

- S3 has no Wave 1 signal, so every non-findings row stays green with a blank Status — silence is the UX.
- The Wave 2 PAB finding is a Broken-style background concern on a Healthy row → `!` glyph, S1 + S3 + S4 + S5.

One row per signal from §3:

| Signal (short) | Wave | State bucket | Severity | Surfaces reached | List text (S4) | Detail text (S5) |
|---|---|---|---|---|---|---|
| `GetPublicAccessBlock` missing or any PAB flag false | 2 | Healthy | `!` | S1, S3, S4, S5 | `public access block incomplete` | `Bucket-level public access block is missing or partial — account-level PAB may still apply.` |

## 4.1 UX review

At 3am, glancing at the list, can the operator tell what's wrong with a problem row without opening detail? Yes for this resource — any bucket flagged with `!` carries `public access block incomplete` in the Status column, which names the concern plainly. The detail line adds the single nuance an operator needs (account-level PAB may override) without forcing a navigation. All problem rows are self-explanatory in the list — operator can triage without opening detail.

## 5. Out of Scope

- All §3.3 Wave 3 signals (copied above): `GetBucketPolicyStatus`, `GetBucketEncryption`, `GetBucketVersioning`, `GetBucketLogging`, `GetBucketLifecycleConfiguration`.
- `iam-user` related-panel entry — discovery mechanism not available in AWS surface for an operator-grade pivot (principal attribution requires CloudTrail data-plane parsing, which is Wave 3). a9s-devops: not worth it.
- `waf` related-panel entry as a first-order S3 pivot — WAF does not attach to S3 directly; the indirect path goes through CloudFront and is already covered by the `cf` panel entry. a9s-devops: not worth it.
- Any UI element not listed in §4 — e.g. new columns, new icons, new views, new key bindings.
- Any write operation. a9s is read-only by design (`architecture.md` §"What is a9s?").
