# r53 ‚Äî Resource Spec

Golden UX/UI doc for this resource, written from the operator's perspective. Describes what the list row, Status column, glyphs, and detail view should look like ‚Äî the should-be, not the is. Implementation conforms to this doc; tests assert against it. When code and this doc disagree, the code is wrong.

## 1. Identity

- **shortName**: `r53`
- **Display name**: Route 53 Hosted Zones
- **AWS API reference**: <https://docs.aws.amazon.com/Route53/latest/APIReference/API_HostedZone.html>
- **List API**: `ListHostedZones` ‚Äî returns `HostedZone[]`. The SDK confirms `Id`, `Name`, `Config.PrivateZone`, `ResourceRecordSetCount`, `LinkedService` are all on the list response shape. Notably, `HostedZone` has no `CreationDate` field ‚Äî the fetcher cannot tell a 2-record zone that's 5 minutes old from one that's 5 years old without a separate call.
- **Describe API (if any)**: `GetDNSSEC` per public zone in Wave 2. `GetHostedZone` is used at detail time for private-zone VPC associations, but is not required for the Wave 1/Wave 2 signals below.

## 2. Related Resources Panel (detail view, right column)

Expected targets from `docs/related-resources.md` Per-type contract: `acm`, `apigw`, `cf`, `elb`, `logs`, `s3`, `vpc`, `ct-events`.

### `acm`

- **Why related**: DNS-validated ACM certificates publish their validation `CNAME`s into this zone ‚Äî when a cert stays in `PENDING_VALIDATION`, the operator comes here to check the record exists.
- **How discovered**: cross-reference the already-loaded `acm` list by matching each cert's `DomainName` (or `DomainValidationOptions[].ResourceRecord.Name` when available from a prior describe) against this zone's `Name` using longest-suffix match ‚Äî a9s-devops: ACM does not reverse-index to hosted zones; suffix-matching the cert's domain against loaded zone names is the idiomatic zero-API-call pivot, and it's the same pivot `acm`'s own spec uses in the opposite direction.
- **Count shown**: yes.

### `apigw`

- **Why related**: API Gateway custom domains are reached via ALIAS `A`/`AAAA` records in this zone pointing at `*.execute-api.<region>.amazonaws.com` (REST) or `d-<id>.execute-api.<region>.amazonaws.com` (HTTP) targets.
- **How discovered**: call `ListResourceRecordSets(HostedZoneId)` once per zone and keep records where `AliasTarget.DNSName` ends in `.execute-api.<region>.amazonaws.com` ‚Äî a9s-devops: there is no reverse index from API Gateway custom domains to hosted zones on either the R53 or APIGW side; enumerating the zone's record set is the only real-world option and operators already expect this to be a per-zone call.
- **Count shown**: yes.

### `cf`

- **Why related**: CloudFront distributions are fronted by ALIAS records in this zone pointing at `*.cloudfront.net` targets ‚Äî when a record breaks, the zone is the first place to look.
- **How discovered**: same `ListResourceRecordSets(HostedZoneId)` scan as `apigw`, keep records where `AliasTarget.DNSName` ends in `.cloudfront.net`. Cross-reference loaded `cf` list by distribution domain name.
- **Count shown**: yes.

### `elb`

- **Why related**: ALB / NLB endpoints are typically exposed under a custom name via an ALIAS record in the zone (`*.elb.<region>.amazonaws.com`, `*.elb.amazonaws.com`, `dualstack.*.elb.<region>.amazonaws.com`).
- **How discovered**: same `ListResourceRecordSets(HostedZoneId)` scan, keep records where `AliasTarget.DNSName` contains `.elb.` ‚Äî a9s-devops: there is no reverse index from a load balancer back to the zone(s) aliasing it; the per-zone record-set scan is the only read-only path, shared with `apigw`/`cf`/`s3` so it's one call per zone, not four.
- **Count shown**: yes.

### `logs`

- **Why related**: Query logging routes every resolved DNS query for this zone into a CloudWatch Logs log group ‚Äî when DNS is behaving unexpectedly, this is where operators go to see the raw queries.
- **How discovered**: call `ListQueryLoggingConfigs(HostedZoneId=<zoneId>)` per zone; the result's `CloudWatchLogsLogGroupArn` is the target `logs` resource ‚Äî a9s-devops: possible=yes, worth=yes. There's no field on `HostedZone` itself pointing at the log group, but the dedicated `ListQueryLoggingConfigs` API is cheap (one call per zone, returns at most one config) and the CloudWatch log group is the primary artifact when investigating query-level DNS issues.
- **Count shown**: yes (0 or 1 per zone).

### `s3`

- **Why related**: S3 static-website endpoints are exposed via ALIAS records in this zone ‚Äî `s3-website.<region>.amazonaws.com` or the older `s3-website-<region>.amazonaws.com` form.
- **How discovered**: same `ListResourceRecordSets(HostedZoneId)` scan, keep records where `AliasTarget.DNSName` contains `s3-website` ‚Äî a9s-devops: possible=yes, worth=yes. The record-set scan is the only way to enumerate which buckets are aliased under this zone; shared with the apigw/cf/elb scan, so no additional API cost.
- **Count shown**: yes.

### `vpc`

- **Why related**: Private hosted zones are associated to one or more VPCs ‚Äî the zone only resolves inside those VPCs, and the operator needs to see which.
- **How discovered**: applies only when `Config.PrivateZone == true`. Call `GetHostedZone(Id=<zoneId>)` once and read `VPCs[].{VPCId, VPCRegion}` ‚Äî a9s-devops: `ListHostedZones` does not return VPC associations even for private zones; `GetHostedZone` is the one-call authoritative source and is already the standard way operators enumerate private-zone VPC bindings.
- **Count shown**: yes (number of VPC associations).

### `ct-events`

- **Why related**: Universal pivot ‚Äî who created the zone, changed records, enabled DNSSEC, added a VPC association.
- **How discovered**: pre-built CloudTrail query scoped to the zone ID (Route 53 CloudTrail events carry the hosted-zone ID as the resource identifier).
- **Count shown**: unknown (CloudTrail queries are windowed; a reliable total isn't available without a separate count call).
- Universal pivot ‚Äî applies to every registered type; see `related-resources.md` ¬ßPolicy.

## 3. Attention / Issues Algorithm

Transcribed from `docs/attention-signals.md`.

### 3.1 Wave 1 ‚Äî zero extra API calls

One bullet per distinct signal. Keep AWS field names verbatim.

- **Signal**: `ResourceRecordSetCount <= 2` on a non-new zone (i.e. the zone carries only the default SOA + NS and nothing else).
  - **State bucket**: Warning.
  - **How obtained**: `HostedZone.ResourceRecordSetCount` from `ListHostedZones`. Note: `HostedZone` has no `CreationDate` on the list response, so "non-new" is not directly observable from Wave 1 alone ‚Äî the signal will also fire on zones created moments ago. See ¬ß4.1 for the UX gap and the operator-facing mitigation.

### 3.2 Wave 2 ‚Äî bounded extra API calls

One bullet per distinct signal.

- **Signal**: DNSSEC in an unhealthy state ‚Äî `GetDNSSEC.Status.ServeSignature == SIGNING` on a zone with `Config.PrivateZone == false` and at least one `KeySigningKeys[].Status != ACTIVE` (e.g. `INACTIVE`, `ACTION_NEEDED`, `INTERNAL_FAILURE`), which means the zone claims to be signing but has no working KSK.
  - **State bucket**: Broken.
  - **API call**: `GetDNSSEC(HostedZoneId)` per public zone (skip when `Config.PrivateZone == true` ‚Äî DNSSEC is not supported on private zones).
  - **Cost shape**: per-resource.

### 3.3 Wave 3 ‚Äî OUT OF SCOPE

- OUT OF SCOPE: Registrar-level nameserver delegation check (does the registrar's NS set match the zone's NS records?) ‚Äî requires an external WHOIS / registrar API; not a read-only AWS call.
- OUT OF SCOPE: Health-check status aggregation per zone ‚Äî would require `ListHealthChecks` + `GetHealthCheckStatus` per health check, multi-call per zone.

## 4. Issue Visualization

Every signal from ¬ß3.1 and ¬ß3.2 must land on one or more of these five existing surfaces. No other UI is allowed.

| # | Surface | Mechanism |
|---|---|---|
| S1 | Menu `issues:N` count | Aggregated count of `!`-severity findings. `~` findings do not bump. |
| S2 | Row color (list view) | Row colored by state bucket ‚Äî Healthy=green, Warning=yellow, Broken=red, Dim=gray. Yellow/red/dim are themselves the attention signal. |
| S3 | `!` / `~` glyph before the name | Annotates a Healthy (green) row with "no immediate action, but worth knowing". **Never appears on yellow/red/dim rows.** |
| S4 | Status / description column text | Short human-readable cause. **Healthy rows render blank.** |
| S5 | Detail view enrichment line | Short operator-readable sentence rendered inline in the detail view. |

Wave ‚Üí surface mapping:

- **Wave 1 Healthy** ‚Üí no ¬ß4 row (omit).
- **Wave 1 Warning / Broken / Dim** ‚Üí S2 + S4.
- **Wave 2 finding on a Healthy row, important** ‚Üí `!` glyph on green row. S1, S3, S4, S5.
- **Wave 2 finding on a Healthy row, informational** ‚Üí `~` glyph on green row. S3, S4, S5. No S1.
- **Wave 2 finding on an already yellow/red/dim row** ‚Üí S3 suppressed, S4 deduplicates with existing cause, S5 carries the full sentence, S1 still counts if `!`.

One row per signal from ¬ß3:

| Signal (short) | Wave | State bucket | Severity | Surfaces reached | List text (S4) | Detail text (S5) |
|---|---|---|---|---|---|---|
| `ResourceRecordSetCount <= 2` | 1 | Warning | n/a | S2, S4 | `only SOA+NS records` | `Zone has only the default SOA and NS records ‚Äî likely unused or still being set up.` |
| DNSSEC `SIGNING` with inactive KSK (public zone) | 2 | Broken | `!` | S1, S3, S4, S5 | `DNSSEC signing broken` | `DNSSEC is enabled but no KSK is ACTIVE ‚Äî resolvers will fail to validate this zone.` |

Notes:

- The DNSSEC-broken finding typically lands on a Wave 1 Healthy (green) row because the zone itself is fine on paper ‚Äî record count is normal, it's the signing key that's dead. That makes it the classic `!`-on-green case: S1 bumps, `!` glyph prefixes the name on the green row, S4 carries the short cause, S5 the full sentence.
- If a zone has BOTH `ResourceRecordSetCount <= 2` (yellow) AND a DNSSEC finding, the row is already yellow so S3 is suppressed; S4 takes whichever cause is more urgent (DNSSEC broken wins), and S5 still carries the DNSSEC sentence from Wave 2.

## 4.1 UX review (two sentences)

At 3am, glancing at the list, can the operator tell what's wrong with a problem row without opening detail? Mostly yes ‚Äî the record-count warning reads plainly as `only SOA+NS records`, and the DNSSEC case carries `DNSSEC signing broken` on the row. The one genuine gap: the record-count signal fires on brand-new zones too (because `HostedZone` carries no `CreationDate` on the list response), so the implementation should suppress the warning when the zone is younger than a short threshold ‚Äî a9s-devops: possible via `GetHostedZone` (not on list) or via a CloudTrail `CreateHostedZone` lookup, but neither fits Wave 1; the pragmatic mitigation is to keep the Wave 1 warning and let operators read it as "zone looks empty, double-check it's not brand-new" rather than pay a per-zone describe just to suppress noise.
