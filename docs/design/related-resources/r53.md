# Route 53 Hosted Zones (r53) — Related Resources

## Real-World Use Cases

**1. "What resources do the records in this zone point to?"** DNS records are the front door to your infrastructure. Alias records point to ELBs, CloudFront distributions, S3 websites, and API Gateways. A/CNAME records point to EC2 IPs and database endpoints. Navigating from DNS → resource is the core workflow.

**2. "Is this zone actually in use?"** Before deleting a hosted zone, verify: are there records beyond the default NS and SOA? Is any domain's DNS delegation pointing to this zone's name servers?

**3. "Why is DNS resolution failing?"** Check if the zone has the expected records, if health checks are passing (for failover routing), and if the NS records in the parent zone match.

## Algorithmic Relationships

| Related Resource | Algorithm | Scenario | Priority |
|-----------------|-----------|----------|----------|
| Records → ELB (elb) | Parse record sets: alias records with `HostedZoneId` matching known ELB hosted zone IDs (e.g., `Z35SXDOTRQ7X7K` for us-east-1 ALB). Alias `DNSName` contains the ELB DNS name. | "Which ELBs does this zone point to?" Core navigation for understanding traffic flow. | P0 |
| Records → CloudFront (cf) | Alias records with `HostedZoneId=Z2FDTNDATAQYW2` (CloudFront's fixed hosted zone ID). | "Which CloudFront distributions does this zone point to?" | P0 |
| Records → S3 Website (s3) | Alias records with hosted zone IDs matching S3 website endpoint zones (vary by region). Or CNAME records pointing to `{bucket}.s3-website-{region}.amazonaws.com`. | "Which S3 buckets serve websites from this zone?" | P1 |
| Records → API Gateway (apigw) | Alias records pointing to API GW custom domain names. Hosted zone ID varies by region. Or CNAME records pointing to `{api-id}.execute-api.{region}.amazonaws.com`. | "Which APIs does this zone point to?" | P1 |
| Records → EC2 / RDS / Redis | A records with IP addresses → match against EC2 public/private IPs. CNAME records → match against RDS endpoints (`{id}.{hash}.{region}.rds.amazonaws.com`), Redis endpoints, DocDB endpoints. | "Which instances and databases does this zone reference?" Critical during decommission — DNS records pointing to deleted resources cause failures. | P1 |
| Health Checks (not in a9s) | `route53:ListHealthChecks` — health checks are referenced by record sets for failover routing. Match health check IDs from record sets to their configurations. | "Why is DNS failing over to the secondary?" The health check on the primary is failing. | P1 |

## CloudTrail Events (T key)

| Event Name | Why Engineers Search For It |
|-----------|---------------------------|
| ChangeResourceRecordSets | "Who changed DNS records?" The most important R53 audit event. DNS changes can redirect traffic to wrong resources or break service discovery. Shows which records were added, changed, or deleted. |
| DeleteHostedZone | "Who deleted this zone?" All records in the zone are lost. Domain becomes unresolvable if NS delegation still points here. |
| CreateHostedZone | "Who created this zone?" Audit trail for DNS infrastructure changes. |
